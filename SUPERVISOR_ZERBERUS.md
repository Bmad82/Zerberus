- **Architektur-Entscheidung (Block 1, K2):** Intent kommt vom **Haupt-LLM via JSON-Header in der eigenen Antwort**, nicht via Regex und nicht via separatem Classifier-Call. Whisper-Transkriptionsfehler machen Regex-Intent-Detection unbrauchbar; ein Extra-Classifier verdoppelt die Latenz für jeden Turn. Das Haupt-LLM liefert beides — Intent + Antwort — in einem Call. Format als allererste Zeile: `{"intent": "CHAT|CODE|FILE|SEARCH|IMAGE|ADMIN", "effort": 1-5, "needs_hitl": bool}`, optional in einem ```json-Code-Fence, danach der eigentliche Body.
- **Drei neue Module:**
  - [`zerberus/core/intent.py`](zerberus/core/intent.py) — `HuginnIntent`-Enum mit den 6 Kern-Intents (CHAT, CODE, FILE, SEARCH, IMAGE, ADMIN). Die 9 weiteren Intents aus dem Review (EXECUTE, MEMORY, RAG, SCHEDULE, TRANSLATE, SUMMARIZE, CREATIVE, SYSTEM, MULTI) sind als Kommentar reserviert für Phase D/E. `HuginnIntent.from_str(value)` mit Fallback auf CHAT bei None/Empty/Unknown — robust gegen kaputte LLM-Outputs.
  - [`zerberus/core/intent_parser.py`](zerberus/core/intent_parser.py) — `parse_llm_response(raw)` mit Brace-Counter statt naivem `[^}]+`-Regex. Robustheit-Garantien: kein Header → Default `(CHAT, 3, False)` und Body = Original; kaputtes JSON → Default + Warning-Log; unbekannter Intent → CHAT; effort außerhalb 1-5 → geclampt; effort nicht-numerisch → 3. Liefert `ParsedResponse(intent, effort, needs_hitl, body, raw_header)`.

Edge-Case „LLM liefert nur Header, kein Body": Roh-Antwort wird gesendet (hässlich, aber besser als leere Telegram-Nachricht).
- **HitL-Policy aktuell (P164-Stand):** Decision wird **geloggt + als Admin-DM-Hinweis** verschickt (Chat-ID + Intent + Effort + Grund). **Patch 163** — Rate-Limiting + Graceful Degradation + Token-Effizienz (2026-04-25)

- **Phase-A-Abschluss (Sicherheits-Fundament).** Schließt die letzten Findings aus dem 7-LLM-Review für Phase A: N3 (kein Per-User Rate-Limit), D1 (Telegram-Ausgangs-Rate-Limit nicht respektiert), K4 (OpenRouter-SPOF ohne Retry/Fallback), O10 (Guard-Fail-Verhalten implizit, nicht konfigurierbar). Außerdem die Querschnitts-Aufgabe „Token-Effizienz" als Block 0 (Doku-Anpassung) — die Bibel-Fibel-Kompression aus Commit `d098738` war der Vorlauf, dieser Patch fügt die ausstehende Lesson und die Regelaktivierung hinzu.
- **Per-User Rate-Limiter (Block 1, N3/D1):** Neue Datei [`zerberus/core/rate_limiter.py`](zerberus/core/rate_limiter.py) mit Interface `RateLimiter` (Rosa-Skelett für Redis-Variante) und `InMemoryRateLimiter` (Huginn-jetzt). Sliding-Window 10 msg/min/User, Cooldown 60s. Singleton via `get_rate_limiter()`. Der Witz steckt im `first_rejection`-Flag: beim ersten Block in einer Cooldown-Periode antwortet Huginn genau einmal mit „Sachte, Keule. Du feuerst schneller als Huginn denken kann. Warte X Sekunden.", danach werden Folge-Nachrichten still geschluckt — kein Reply-Spam. Integration in [`process_update()`](zerberus/modules/telegram/router.py) ganz oben (nach Update-Typ-Filter, vor Event-Bus), nur für `message`-Updates, Callback-Queries (Admin-HitL-Klicks) sind ausgenommen. `cleanup()` entfernt Buckets nach 5 Minuten Inaktivität.
- **Guard-Fail-Policy (Block 2, K4/O10):** Neuer Config-Key `security.guard_fail_policy` (Default `allow`, Werte `allow`/`block`/`degrade`). Helper `_resolve_guard_fail_policy(settings)` in [`router.py`](zerberus/modules/telegram/router.py) liest den Key aus dem `security`-Top-Level-Dict (via `extra="allow"` in der Pydantic-Settings-Klasse). Bei Guard-Verdict `ERROR`: `allow` lässt die LLM-Antwort durch und loggt eine Warnung (Huginn-Modus, bisheriges Verhalten); `block` blockiert die Antwort und sendet stattdessen „⚠️ Sicherheitsprüfung nicht verfügbar. Antwort zurückgehalten." (Rosa-Modus); `degrade` ist als Pfad reserviert, fällt aktuell auf `allow` zurück (Future: Ollama-Fallback). Beide Pfade — `_process_text_message` (DMs + direkte Gruppen-Ansprache) und der autonome Gruppen-Einwurf — respektieren die Policy.
- **OpenRouter-Retry mit Backoff (Block 2, O10):** Neuer Wrapper `_call_llm_with_retry()` um `call_llm()`. `call_llm` raised nicht — Fehler kommen als `{"content": "", "error": "HTTP 429"}` zurück. Der Wrapper erkennt retryable Fehler (`429`/`503`/„rate" im Error-String) und wartet 2s/4s/8s zwischen den Versuchen (max 3). 400/401/etc. werden sofort durchgereicht — Bad Request bleibt Bad Request. Bei Erschöpfung in der DM-Pipeline: Fallback-Antwort „Meine Kristallkugel ist gerade trüb. Versucht's später nochmal. 🔮". Im autonomen Gruppen-Einwurf: still überspringen (niemand hat gefragt).
- **Telegram-Ausgangs-Throttle (Block 3, D1):** Neuer Helper `send_telegram_message_throttled()` in [`bot.py`](zerberus/modules/telegram/bot.py). Pro Chat-ID werden ausgehende Timestamps der letzten 60s getrackt. Bei Überschreitung von 15 msg/min (konservativ unter Telegrams ~20 msg/min/Gruppe) wartet die Funktion via `asyncio.sleep`, bis das älteste Fenster-Element rausfällt — Nachricht wird nicht gedroppt. Aktuell genutzt im autonomen Gruppen-Einwurf-Pfad. DMs (privat) bleiben bei `send_telegram_message` direkt, da dort kein Gruppen-Limit greift.
- **Token-Effizienz (Block 0):** Die neue Sektion in `CLAUDE_ZERBERUS.md` aus Commit `d098738` ist aktiv (keine rituellen File-Reads, ein Read→Write-Zyklus pro Datei am Patch-Ende). Diese Lesson ist jetzt auch in `lessons.md` als „Token-Effizienz bei Doku-Reads (P163)" eingetragen — und die neue Lesson „Rate-Limiting + Graceful Degradation (P163)" dokumentiert die wichtigsten Entscheidungen (Singleton, `first_rejection`, Retry nur bei 429/503, Throttle wartet statt droppt, Config-Keys vorbereitet aber Defaults aktiv).
- **Config-Keys vorbereitet, nicht aktiv gelesen:** `limits.per_user_rpm`/`limits.cooldown_seconds`/`security.guard_fail_policy` sind in `config.yaml` eingetragen. Aktiv gelesen wird **nur** `security.guard_fail_policy` (durch den Helper `_resolve_guard_fail_policy`). Die `limits.*`-Werte sind als Hooks für Phase B (Config-Refactor) vorbereitet — bis dahin steuern die Defaults aus dem Code (max_rpm=10, cooldown=60). Das verhindert, dass wir jetzt schon Pydantic-Modell-Erweiterungen für eine erst Phase-B-relevante Konfiguration einbauen.
